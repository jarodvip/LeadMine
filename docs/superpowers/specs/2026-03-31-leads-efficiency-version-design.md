# 2026-03-31 线索管理提效版设计

## Summary

本版本聚焦“线索管理提效”，不扩大发现链路，不新增重型 CRM 能力。目标是在现有采集、处理、线索入库主流程已可用的基础上，优先提升销售处理已入库线索的效率：更快筛选、更快批量处理、更快导出，并形成最小可用的跟进闭环。

## Goal

本版本要实现以下业务结果：

- 让用户在 Leads 列表中更快完成高频处理动作。
- 让用户能够基于当前筛选条件导出线索结果，便于外部二次处理。
- 让用户以更低操作成本完成批量状态更新和批量分配。
- 让线索从“发现”到“分配/跟进”具备最小闭环。

成功标准：

- 支持按筛选条件稳定获取线索结果。
- 支持当前页选中线索的批量状态更新。
- 支持当前页选中线索的批量分配。
- 支持按当前筛选条件导出标准字段结果。
- 支持单条线索维护基础备注。
- 列表中可见负责人、状态、关键时间等辅助处理字段。

## Non-goals

本版本明确不做：

- 不新增大批采集来源。
- 不重做 NLP、去重或识别算法。
- 不建设完整 CRM、审批流或客户档案体系。
- 不引入复杂消息中心、自动提醒或组织级权限系统重构。
- 不承诺“按整个筛选结果集批量处理”，首版只保证“当前页已选项”语义。

## Current Context

结合当前仓库状态与近期主线提交，可确认以下事实：

- 微信 RSSHub 来源支持已完成。
- Leads 当前页批量操作已完成。
- Leads 首屏 keyword 初始化加载问题已修复。
- 前后端最小验证入口与交付质量文档正在收口。

这说明项目现阶段的主要矛盾不再是“是否具备线索发现能力”，而是“现有线索能否被更高效地处理和流转”。因此，下个版本应优先从线索管理端提效，而不是继续扩大发现端范围。

## Approaches Considered

### A. 处理效率优先（推荐）

优先建设导出、批量状态流转、批量分配和列表快捷处理。

优点：

- 业务收益出现最快。
- 范围边界清晰，适合一个版本内闭环。
- 与“处理效率提升”这一主目标最一致。

缺点：

- 对“找线索更快”的帮助有限。
- 跟进闭环只会做到最小版本。

### B. 检索效率优先

优先建设更强筛选、搜索、排序和列表视图组织。

优点：

- 当线索量上升后，用户体感会明显改善。
- 更适合为后续精细化运营铺路。

缺点：

- 对处理吞吐量的直接提升不如方案 A。
- 容易扩展为复杂查询体验重构。

### C. 跟进闭环优先

优先建设分配、备注、状态流转和负责人视图。

优点：

- 更接近销售协作链路。
- 团队协同价值较高。

缺点：

- 容易把版本做重。
- 容易引出权限、审计、日志流等额外范围。

## Selected Strategy

采用 **A 为主，B 为辅，C 做最小闭环** 的版本策略。

优先级排序如下：

### P0：处理效率

1. 线索导出
2. 批量状态更新补强
3. 批量分配补强
4. 列表快捷处理优化

### P1：检索效率

1. 筛选条件增强
2. 搜索体验优化
3. 排序能力优化

### P2：最小跟进闭环

1. 基础备注能力
2. 负责人维度可见
3. 最小状态流转闭环

## Proposed Design

### 1. 以 Leads 列表页为主战场

本版本不新起独立子系统，直接围绕 Leads 列表页进行增量扩展。

设计原则：

- 能在列表完成的操作，不强制进入详情页。
- 能复用现有接口和字段的，不先做底层重构。
- 能用简单明确交互完成的，不引入多层弹窗或复杂流程。

这意味着前端改动重心会集中在列表页操作区、筛选区、表格列和轻量编辑入口。

### 2. 导出能力

导出能力以“按当前筛选条件导出标准字段结果”为边界。

首版优先 CSV：

- 不引入更重的 Excel 依赖与格式兼容问题。
- 适合做最小可行版本。
- 可满足用户在表格工具中继续处理的核心诉求。

导出内容只覆盖标准列表字段，不做复杂模板、字段自定义或异步导出任务中心。

### 3. 批量处理能力

复用现有“当前页批量操作”语义，增强以下动作：

- 批量状态更新
- 批量分配

交互上需给出明确反馈，至少要让用户知道：

- 本次处理了多少条。
- 是否有失败项。
- 刷新后当前列表状态是否已同步。

本版本不把语义扩展为“对全部筛选结果批量处理”，避免误用和范围膨胀。

### 4. 检索效率能力

检索优化围绕高频使用维度展开：

- 状态
- 事件类型
- 时间范围
- 来源
- 关键词
- 排序

默认视图保持简单，不引入保存视图、复杂组合面板或个性化布局。排序优先考虑直接支撑处理效率的字段，如发布时间、创建时间、置信度等。

### 5. 最小跟进闭环

本版本只做轻量闭环：

- 支持基础备注维护。
- 支持在列表中看到负责人。
- 支持围绕现有状态枚举完成基础流转。

备注首版不做完整操作日志、评论流或历史版本。负责人维度只要求“可见、可筛、可更新”，不引入更复杂的组织协作模型。

## Data and API Impact

### Existing data reuse

根据当前技术设计文档，`leads` 已具备以下字段：

- `status`
- `assigned_to`
- `sales_notes`
- `published_at`
- `confidence`

因此本版本优先复用现有数据结构，不预设大规模表结构调整。若实现时发现某一能力确需补字段，应以最小增量为原则。

### API impact

建议围绕现有 leads 相关接口扩展：

- `GET /api/v1/leads`
  - 扩展筛选与排序参数。
- `PATCH /api/v1/leads/{id}`
  - 支持备注、状态、负责人等单条更新。
- 新增批量接口
  - 用于批量状态更新。
  - 用于批量分配。
- 新增导出接口
  - 用于按当前筛选条件返回导出文件。

接口边界原则：

- 单条编辑继续走单条接口。
- 批量动作使用独立接口。
- 导出使用独立接口，不把文件下载混入列表查询接口。

## User Flows

### 场景 A：筛选并批量处理

1. 用户进入 Leads 列表。
2. 设置筛选条件、关键词和排序方式。
3. 选择当前页线索。
4. 执行批量状态更新或批量分配。
5. 前端刷新当前列表并反馈结果。

### 场景 B：按筛选结果导出

1. 用户设置筛选条件。
2. 点击导出。
3. 后端按当前条件生成导出结果。
4. 浏览器下载文件。
5. 用户在外部表格工具中继续处理。

### 场景 C：单条补备注形成闭环

1. 用户在列表或详情进入备注操作。
2. 保存备注。
3. 后端更新线索记录。
4. 列表继续显示负责人、状态、关键时间等字段，支持后续流转。

## Error Handling

只处理真实边界场景：

- 批量操作返回成功/失败摘要。
- 导出结果为空时返回清晰提示。
- 非法筛选参数由后端显式校验并返回明确错误。
- 不为理论上不会发生的内部状态增加过度兜底逻辑。

## Testing Strategy

### Automated

后端：

- 补 leads 查询参数测试。
- 补批量状态更新测试。
- 补批量分配测试。
- 补导出接口测试。

前端：

- 补 Leads 列表页筛选交互测试。
- 补批量操作交互测试。
- 补导出入口测试。
- 补备注相关交互测试。

### Acceptance

版本验收以以下结果为准：

1. 用户可按条件筛选并稳定获取结果。
2. 用户可对当前页选中线索执行批量状态更新。
3. 用户可对当前页选中线索执行批量分配。
4. 用户可按当前筛选条件导出标准字段结果。
5. 用户可对单条线索维护基础备注。
6. 负责人、状态、关键时间字段在列表中可见并可辅助处理。

## Risks and Controls

### 风险 1：版本范围失控

**控制：** 备注只做基础能力，不引入完整日志流或协作流。

### 风险 2：导出诉求快速膨胀

**控制：** 首版只支持标准字段、按当前筛选条件导出，不做字段配置与异步导出中心。

### 风险 3：前端交互复杂度上升

**控制：** 优先列表直达动作，不引入过多弹窗链路。

### 风险 4：批量语义不清导致误操作

**控制：** 首版只保证“当前页已选项”语义，界面与接口文案都要明确表达这一点。

## Out of Scope Follow-up

适合作为后续版本候选，但不纳入本版：

- 全量筛选结果批量处理
- 自定义导出字段
- 异步导出任务中心
- 完整备注历史与协作评论流
- 组织级权限、审批、提醒、通知中心
- 更大范围的来源扩张与识别模型优化

## Rollout Result

本版本完成后，LeadMine 将从“已有线索管理功能”进一步提升到“可高频处理和流转线索”的阶段。用户能更快找到线索、批量处理线索、导出线索，并用最小成本完成基础跟进闭环，为后续更重的销售协作能力打下更稳的使用基础。
