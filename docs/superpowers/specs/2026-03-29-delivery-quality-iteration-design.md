# 2026-03-29 交付质量收口迭代设计

## Summary

本迭代只做“交付质量收口”，不新增业务功能、不改变现有产品行为。目标是把当前已合入主线的一批能力整理成一个更易验证、易交付、易说明的稳定基线：补齐前端标准测试入口、固化仓库最小验证命令、整理一版可直接复用的发布说明，并修正关键文档中与当前代码状态不一致的内容。

## Goal

让仓库当前状态满足以下要求：
- 前端测试有标准入口，不再依赖临时命令记忆。
- 前后端最小验证方式明确且已实际可运行。
- 最近一轮已交付内容有一版清晰、可复用的中文发布说明。
- 关键产品/计划/技术文档与当前代码能力保持一致。

## Non-goals

- 不新增来源管理功能。
- 不增强 Leads 新功能。
- 不改动现有业务流程或接口行为。
- 不引入 CI/CD、自动化发布或新的工程化体系。
- 不做大规模文档重写，只修正与现状明显不一致的部分。

## Current Problem

当前仓库已经具备最近一轮主线能力：
- 微信 RSSHub 来源支持已合入。
- Leads 当前页批量操作已合入。
- Leads 首屏 keyword 初始化请求问题已修复。
- 前后端测试已能在本地跑通。

但交付层面仍有几个割裂点：
- `web` 目录下可通过 `npx vitest run` 跑测试，但 `web/package.json` 没有标准 `test` 脚本。
- 仓库的最小验证入口没有被统一表达，收尾时仍需要重新确认命令。
- 最近一轮功能虽已完成，但缺少一份面向交付的简明发布说明。
- 关键文档可能仍停留在旧状态，需要与当前代码事实重新对齐。

## Proposed Design

### 1. 统一前端测试入口

在 `web/package.json` 中增加标准 `test` 脚本，使当前 Vitest 执行方式通过包脚本暴露出来。

预期行为：
- 在 `web` 目录执行 `npm test` 即可运行现有前端测试。
- 现有 Vitest 用例继续保持不变。
- 不新增新的测试框架封装。

为什么这样做：
- 降低验证入口的记忆成本。
- 让前端验证方式与常规 Node 项目习惯一致。
- 后续文档、交付说明、计划都可以稳定引用同一个命令。

### 2. 固化仓库最小验证基线

将本仓库当前最小可交付验证动作收敛为两条：
- 后端：`cd backend && python3 -m pytest`
- 前端：`cd web && npm test`

这一迭代不引入新的统一根脚本，也不额外包装复杂验证命令。

为什么这样做：
- 当前仓库前后端目录结构清晰，分别执行命令已经足够直接。
- 保持最小变更，不为了“更统一”引入新的脚本层。
- 后续如果需要再演进为根目录统一命令，也应在独立迭代处理。

### 3. 输出一版交付发布说明

基于最近已经进入主线的内容，整理一版可直接复用的中文发布说明。说明范围限定为已完成且已验证内容：
- 微信 RSSHub 来源支持
- Leads 批量操作
- Leads 首屏 keyword 初始化修复
- 测试覆盖补强
- 文档目录整理

发布说明的目标不是重新写方案，而是把“已经交付了什么”清晰说出来。

预期结果：
- 可直接用于 PR 描述、版本说明或内部同步。
- 内容只覆盖已落地事实，不承诺未做事项。

### 4. 对齐关键文档与当前状态

检查以下文档，并只修正与当前代码状态明显不一致的内容：
- `docs/PRD.md`
- `docs/开发计划.md`
- `docs/技术方案设计.md`

修订原则：
- 只做现状对齐。
- 不借机扩写成新一轮需求设计。
- 若文档中的未来计划仍然合理但尚未实现，应明确其为规划状态，而不是已完成状态。

## Execution Order

### Step 1. 补前端测试脚本
先修改 `web/package.json`，把当前可用的 Vitest 命令正式暴露为标准脚本。

验收：
- `cd web && npm test` 能跑通现有前端测试。

### Step 2. 确认仓库验证基线
在前端测试脚本补齐后，明确本仓库当前交付前的最小验证动作。

验收：
- `cd backend && python3 -m pytest` 可运行。
- `cd web && npm test` 可运行。
- 文档/说明中引用的验证命令与实际一致。

### Step 3. 整理发布说明
在验证基线明确后，整理一版中文发布说明。

验收：
- 说明范围与最近主线实际改动一致。
- 内容能直接复用，不需要再次大幅改写。

### Step 4. 校正文档
最后对关键文档进行现状对齐，避免边改命令边返工文档。

验收：
- 文档内容与当前仓库能力一致。
- 修改范围可控，没有扩展为新的需求设计工作。

## Testing Strategy

### Automated
本迭代的自动化验证以现有测试为准：
1. 后端运行 `cd backend && python3 -m pytest`
2. 前端运行 `cd web && npm test`

重点不是新增测试，而是确保验证入口稳定、表达一致。

### Manual
人工检查：
1. 查看 `web/package.json` 是否存在标准 `test` 脚本。
2. 确认交付说明覆盖最近已完成内容。
3. 抽查关键文档，确认已完成事项、规划事项、未完成事项没有混写。

## Trade-offs

### Benefits
- 范围小，风险低。
- 不触碰业务行为，适合作为当前主线功能的收口迭代。
- 能明显降低后续每轮开发和验收的沟通成本。
- 为下一迭代功能开发建立更稳定的交付基线。

### Limitations
- 用户不会立即感知到新功能。
- 主要收益体现在交付质量、可维护性和协作效率上。
- 不解决后续功能优先级问题，只是为下一轮开发做铺垫。

## Alternatives Considered

### A. 直接进入来源管理/抓取链路新功能
未采用，因为当前主线虽然功能已完成，但交付层仍存在测试入口与说明不统一的问题，继续叠加功能会放大收尾成本。

### B. 直接进入 Leads 能力增强
未采用，因为 Leads 近期已经连续完成多项改动，此时更适合先收口验证与文档基线，再继续叠加需求。

### C. 一次性同时推进质量收口、来源链路和 Leads 增强
未采用，因为这会让迭代边界重新变大，降低可预测性，不符合本轮“只做交付质量”的明确选择。

## Rollout Result

完成本迭代后：
- 前端测试入口与实际测试方式一致。
- 仓库前后端最小验证命令明确且可执行。
- 最近主线功能有一版可复用的交付说明。
- 关键文档与当前代码状态保持一致。
- 下一轮功能迭代可以在更稳定的交付基线上开始。

## Out of Scope Follow-up

适合作为下一迭代候选但不在本次范围内：
- 来源管理与抓取调度体验优化
- Leads 筛选、导出与批量操作进一步增强
- 根目录统一验证脚本或 CI 自动化接入
