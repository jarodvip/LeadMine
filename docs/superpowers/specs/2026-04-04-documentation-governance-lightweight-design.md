# 2026-04-04 LeadMine 文档治理轻量改造设计

## Summary

本设计采用方案 A（轻治理优先），在不大规模重写历史文档的前提下，解决 LeadMine 当前文档体系中“状态失真、入口分散、计划失效、职责混乱”的问题。核心思路是以最小改动建立统一入口与滚动进度页，并为核心旧文档补充明确定位，让文档体系重新具备可读、可信、可维护的基础。

## Goal

本轮文档治理要达成以下结果：

- 用户能够通过一份统一入口文档快速了解项目现状。
- 当前开发进度有独立、稳定的表达位置。
- 历史规划文档不再被误读为实时开发状态。
- 产品文档、计划文档、技术文档与迭代文档边界清晰。
- 后续每轮迭代只需维护少量固定文档，降低漂移风险。

## Non-goals

本轮明确不做：

- 不重写全部历史文档。
- 不大规模重命名、归档或删除现有文档。
- 不改变业务需求范围或技术实现范围。
- 不重写 `docs/superpowers/specs/`、`plans/`、`release-notes/` 中已有迭代档案。
- 不引入复杂的文档自动化、流程平台或治理体系。

## Current Problem

结合当前仓库中的 `docs/PRD.md`、`docs/开发计划.md`、`docs/技术方案设计.md` 与近期 `docs/superpowers/` 下的迭代设计/计划文档，可以确认当前文档体系存在以下问题：

### 1. 状态失真

核心功能主链路已经具备可运行版本，近期还完成了微信 RSSHub 来源支持、Leads 当前页批量操作、Leads 列表/导出参数一致性、批量反馈优化以及回归测试补强。但 `docs/开发计划.md` 仍以早期周计划方式组织内容，容易让读者误以为项目仍停留在早期阶段。

### 2. 入口分散

项目当前状态散落在 PRD、开发计划、技术方案、迭代 spec、实施 plan 和 release note 中。用户无法在一个稳定入口中快速获得“现在做到哪”的答案。

### 3. 文档职责混乱

PRD、开发计划、技术方案与迭代文档之间存在表达重叠：

- PRD 既写目标，也隐含了部分落地判断。
- 开发计划既像初版排期，也被拿来理解当前状态。
- 技术方案既描述现状，又混入偏理想化的未来蓝图。
- 迭代 spec/plan 已经承担真实变更描述，但主文档并未与之形成清晰分层。

### 4. 维护成本高

当前文档没有明确的“单一事实源”，导致任何一次功能交付都可能需要同步多处文字说明，而这些说明彼此并不总是一致。

## Design Principles

本轮文档治理遵循以下原则：

1. **轻治理优先**：先解决最影响理解与维护的问题，不做全文重构。
2. **单一事实源**：同一类信息只在一个固定位置持续维护。
3. **文档分层**：一份文档只解决一种核心问题。
4. **现状优先**：文档应以当前仓库与已交付行为为准，而非理想化表述。
5. **增量修正**：只修正与现状明显冲突的内容，不借机扩写需求。

## Proposed Design

### 1. 建立统一入口：`docs/项目现状总览.md`

新增 `docs/项目现状总览.md` 作为项目统一入口与单一事实源，用于集中表达以下信息：

- 项目当前状态的一句话总结。
- 当前已交付能力清单。
- 当前阶段重点。
- 下一阶段候选项。
- 当前验收基线。
- 关键文档入口链接。

该文档只做“导航 + 状态板”，不承载详细需求、详细架构或单轮实施步骤。目标是让用户在 1 分钟内理解项目当前所处阶段。

建议内容结构：

```markdown
# 项目现状总览

## 项目当前状态
## 当前已交付
## 当前阶段
## 下一阶段候选
## 当前验收基线
## 关键文档入口
```

### 2. 建立滚动进度页：`docs/当前开发计划.md`

新增 `docs/当前开发计划.md` 作为实时进度表达页，只维护以下内容：

- 当前阶段。
- 已完成事项。
- 进行中事项。
- 下一步计划。
- 风险与阻塞。
- 当前验收方式。

这样可以把“当前做到哪”的表达从旧 `docs/开发计划.md` 中剥离出来，使历史计划文档回归原本角色。

建议内容结构：

```markdown
# 当前开发计划

## 当前阶段
## 已完成
## 进行中
## 下一步计划
## 风险与阻塞
## 验收方式
```

### 3. 给旧核心文档补充定位说明

对以下文档进行最小改动，在开头补充统一风格的元说明：

- `docs/PRD.md`
- `docs/开发计划.md`
- `docs/技术方案设计.md`

推荐增加的三个块：

```markdown
## 文档定位
## 当前状态说明
## 最后校准时间
```

各文档的预期定位如下：

#### `docs/PRD.md`
- 只描述产品目标、功能范围、优先级与成功指标。
- 不承担实时开发完成度表达。
- 若需提及落地情况，仅做轻量状态说明，并指向《当前开发计划》。

#### `docs/开发计划.md`
- 保留为项目初版阶段规划与路线图。
- 不再作为实时开发进度页维护。
- 文档开头显式提示读者：当前实际进度请参考《当前开发计划》。

#### `docs/技术方案设计.md`
- 描述当前实现架构与关键技术边界。
- 需显式区分“已实现能力”与“规划中/扩展方向”。
- 避免以理想化整体蓝图替代当前仓库现状。

### 4. 小范围校正与现状冲突的表述

在不重写全文的前提下，对核心文档中与当前项目状态明显冲突的内容做小范围修正，重点包括：

- 已实现能力仍写成规划项。
- 未实现能力写得像已交付项。
- 技术方案中对当前系统形态的描述明显偏离仓库结构。

例如：

- `docs/开发计划.md` 中继续使用“第 1 周 ~ 第 6 周”表达历史节奏，但不再默认其代表当前阶段状态。
- `docs/技术方案设计.md` 中需要把当前实现与未来扩展方向区分开，而不是让读者默认所有结构都已完整落地。

### 5. 维持现有迭代档案结构，不纳入本轮重构

`docs/superpowers/specs/`、`docs/superpowers/plans/` 与 `docs/superpowers/release-notes/` 已经承担单轮迭代设计、实施计划与交付说明职责。本轮不重写这些档案，只在新的总览/当前计划文档中建立入口和引用关系。

## Information Architecture

本轮治理完成后的文档层级如下：

- `docs/项目现状总览.md`
  - 统一入口 / 单一事实源
- `docs/当前开发计划.md`
  - 滚动进度表达
- `docs/PRD.md`
  - 产品目标、范围、优先级
- `docs/开发计划.md`
  - 历史阶段规划 / 初版路线图
- `docs/技术方案设计.md`
  - 当前实现架构与技术边界
- `docs/superpowers/specs/*.md`
  - 单轮迭代设计
- `docs/superpowers/plans/*.md`
  - 单轮实施计划
- `docs/superpowers/release-notes/*.md`
  - 已交付内容说明

## File Changes

### New files

- `docs/项目现状总览.md`
- `docs/当前开发计划.md`

### Modified files

- `docs/PRD.md`
  - 增加文档定位、状态说明和校准时间。
  - 仅修正少量与现状冲突的表述。
- `docs/开发计划.md`
  - 增加“历史规划”定位说明。
  - 明确将实时进度引导到《当前开发计划》。
- `docs/技术方案设计.md`
  - 增加定位说明。
  - 校正当前实现与扩展设计的边界。

### Explicitly unchanged

- `docs/superpowers/specs/*.md`
- `docs/superpowers/plans/*.md`
- `docs/superpowers/release-notes/*.md`

## Rollout Order

建议按以下顺序实施：

1. 新增 `docs/项目现状总览.md`。
2. 新增 `docs/当前开发计划.md`。
3. 为 `PRD.md`、`开发计划.md`、`技术方案设计.md` 增加定位说明。
4. 对明显冲突内容做小范围状态校正。

这样做可以先解决“从哪看”和“当前做到哪”的问题，再收敛旧文档语义，避免陷入全文重写。

## Governance Rules

实施完成后，后续维护遵循以下规则：

1. `docs/项目现状总览.md` 始终作为项目第一入口。
2. 当前进度只在 `docs/当前开发计划.md` 维护。
3. `docs/开发计划.md` 只保留历史规划用途。
4. 每轮重要迭代结束后，优先更新：
   - `docs/项目现状总览.md`
   - `docs/当前开发计划.md`
   - 对应 `docs/superpowers/release-notes/*.md`
5. `docs/PRD.md` 不写日报式状态，不承担进度汇总职责。
6. `docs/技术方案设计.md` 必须区分“已实现 / 规划中”。

## Testing and Review Strategy

本轮属于文档治理，不涉及业务行为变更。验收以人工检查为主：

1. 打开 `docs/项目现状总览.md`，确认可在 1 分钟内理解项目现状。
2. 打开 `docs/当前开发计划.md`，确认可读出当前阶段、已完成与下一步。
3. 打开 `docs/开发计划.md`，确认读者不会再把它误读为实时计划。
4. 打开 `docs/技术方案设计.md`，确认当前实现与规划能力边界清晰。
5. 抽查文档跳转关系，确认读者可从总览页进入其他核心文档。

## Benefits

- 以最小改动快速恢复文档可信度。
- 为项目建立稳定入口，降低沟通和交接成本。
- 将历史规划与当前进度解耦，减少误读。
- 为后续迭代沉淀形成低成本更新路径。

## Trade-offs

### Benefits of this approach

- 实施成本低，适合当前阶段快速落地。
- 不破坏已有文档沉淀。
- 能优先解决最影响协作的问题。

### Limitations

- 历史文档结构仍然会保留部分旧痕迹。
- 不是一次性彻底重构，需要后续继续增量治理。
- 依然依赖人工维护，不会自动防止未来漂移。

## Alternatives Considered

### A. 中度重构

在本方案基础上，重写 `docs/开发计划.md` 与 `docs/技术方案设计.md`，并统一梳理交叉引用。

未选原因：改动面更大，容易超出当前必要范围。

### B. 全量重建文档体系

重新设计 docs 信息架构，迁移或归档旧文档。

未选原因：成本和风险都过高，不符合“轻治理优先”的目标。

## Success Criteria

本设计实施完成后，应满足：

- 用户可通过一份入口文档快速获得项目现状。
- 当前开发进度有独立且稳定的表达位置。
- 历史阶段规划不再被误读为实时计划。
- 技术方案不会再把规划能力与现状混写。
- 后续每轮迭代只需更新少量固定文档即可维持一致性。
