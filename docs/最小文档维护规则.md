# 最小文档维护规则

## 文档定位

本文档用于约束 LeadMine 每轮迭代后的最小文档同步动作，以最低维护成本维持 `docs/项目现状总览.md`、`docs/当前开发计划.md` 与仓库实现一致。

## 适用场景

当以下任一事项发生变化时，本轮交付结束前应执行文档同步：

- 新增、删除或明显调整用户可感知能力。
- 当前阶段、进行中事项、下一步安排发生变化。
- 默认本地运行基线、验收口径或关键依赖发生变化。
- 技术方案中“已实现 / 规划中”边界发生变化。
- 新增一份需要长期引用的 spec、plan 或 release note。

## 最小更新范围

### 必更文档

- `docs/当前开发计划.md`
  - 更新当前阶段、已完成、进行中、下一步计划、风险与阻塞中的受影响项。
- `docs/项目现状总览.md`
  - 更新当前已交付、当前阶段、当前验收基线、关键文档入口中的受影响项。

### 按需更新文档

- `docs/技术方案设计.md`
  - 仅在当前实现架构、默认运行基线、关键接口或“已实现 / 规划中”边界变化时更新。
- `docs/PRD.md`
  - 仅在产品目标、范围、优先级或成功指标变化时更新；不写日报式进度。
- `docs/开发计划.md`
  - 仅在需要补充历史背景或定位提示时更新；不作为实时状态页维护。
- `docs/superpowers/release-notes/*.md`
  - 需要保留本轮交付说明时补充或新增。

## 每轮迭代后的执行顺序

1. 先判断本轮变更是否影响用户可见能力、当前阶段、运行基线或技术边界。
2. 先更新 `docs/当前开发计划.md`，再更新 `docs/项目现状总览.md`。
3. 若变更影响技术边界或产品范围，再补改 `docs/技术方案设计.md` 或 `docs/PRD.md`。
4. 若本轮交付需要独立沉淀说明，再补 `docs/superpowers/release-notes/*.md` 并把入口挂到 `docs/项目现状总览.md`。
5. 提交前人工复核一次，确认文档没有把规划写成已实现，也没有遗漏当前阶段变化。

## 变更判定口径

| 变更类型 | 必改文档 | 说明 |
| --- | --- | --- |
| 新增或删除用户可感知能力 | `当前开发计划`、`项目现状总览` | 例如 Leads 批量操作、导出、详情维护能力变化 |
| 当前阶段或下一步变化 | `当前开发计划`、`项目现状总览` | 两份文档都要同步阶段表述 |
| 运行基线或验收口径变化 | `项目现状总览`，必要时 `技术方案设计` | 如默认 Compose 组件、测试基线变化 |
| 技术边界变化 | `技术方案设计`，必要时 `项目现状总览` | 必须明确区分已实现与规划中 |
| 产品目标或范围变化 | `PRD`，必要时 `项目现状总览` | 不在 `PRD` 中写实时进度 |
| 历史参考定位变化 | `开发计划` | 保持历史定位，不回填实时状态 |

## 人工验收清单

- `docs/项目现状总览.md` 仍可在 1 分钟内说明项目现状与入口。
- `docs/当前开发计划.md` 可独立表达当前阶段、已完成事项与下一步。
- `docs/PRD.md` 不承担实时进度汇总职责。
- `docs/技术方案设计.md` 没有把规划能力写成当前默认实现。

## 禁止事项

- 只改代码，不同步 `项目现状总览` 与 `当前开发计划` 中受影响内容。
- 在 `PRD.md` 中写日报式或周报式进度。
- 在 `技术方案设计.md` 中混写当前实现与规划中能力，导致读者误判实际状态。
- 把 `开发计划.md` 再次当作实时滚动计划页维护。
